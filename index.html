Chapter 01

Arrays — The Hostel of Fixed Rooms

Access O(1) Insert/Delete O(n)

The Grand Array Hostel stood as a testament to order and precision. Built from the ground up, its blueprint dictated an immutable structure of perfectly uniform rooms, each assigned an address — its index — starting with the venerable Room 0. The rooms were not just numbered; they were physically contiguous, bolted side-by-side on a single, uninterrupted stretch of memory land.

This rigidity granted the Master a unique power: Instantaneous Access. If the Master needed Room 3, there was no search, no delay. By the simple calculation of Base Address + 3 steps, the door opened — a promise of O(1) retrieval speed. You could find any guest, whether in Room 1 or Room 1,000,000, in the blink of an eye.

However, life was defined by static nature. The real drama unfolded whenever a resident needed to be inserted mid-way. The rule was absolute: all subsequent residents must shift. The guest in Room 2 moves to 3, the one in 3 to 4 — all the way to the end. This mass relocation defined the Hostel's major weakness: a grueling O(n) operation, where effort grew directly proportional to the size of the Hostel.

💡 Analogy: An array is like lockers in a corridor — fixed count, fixed position, instant access by number.

In-Depth Learnings

  • Memory: Static Memory Allocation — size is fixed at initialization and cannot change.
  • Storage: Elements are stored contiguously in memory, enabling direct address calculation.
  • Access: Retrieval by index is O(1) (Constant Time) — the primary performance advantage.
  • Insertion/Deletion: Requires shifting up to N elements — O(n) (Linear Time).
  • Use Cases: Best when collection size is predictable, lookups are frequent, and modifications are rare.

Chapter 02

Linked Lists — The Train with Connectable Cars

Access O(n) Insert/Delete O(1)

Meet the Linked List Train, born from the desire for flexibility and freedom. Unlike the Array Hostel's fixed structure, the Linked List is a collection of independent, movable units known as Nodes. These nodes are scattered across the memory landscape wherever a free spot can be found — they are non-contiguous.

What holds the train together is not physical proximity, but an unwavering sense of direction. Each Node carries its Data and a Pointer — an address label that points precisely to the location of the next car. The whole chain begins with a single designated starting point: the Head.

Adding a car after Node B requires only two quick actions: change B's pointer to the new car, and the new car's pointer to C. No mass movement, no relocations. This is O(1). However, finding the tenth car means starting at the Head and following every pointer sequentially — a slow O(n) operation.

💡 Analogy: A linked list is like a Scavenger Hunt — each clue points to the next location. You can't skip ahead; you must follow every step.

In-Depth Learnings

  • Memory: Dynamic Memory Allocation — the list can fluidly expand or contract during execution.
  • Storage: Nodes are allocated non-contiguously; logical order is maintained exclusively by internal reference pointers.
  • Access: No direct access — finding the Nth element requires traversal from the head: O(n).
  • Insertion/Deletion: Once the insertion point is identified, only a constant number of pointer changes are needed: O(1).
  • Node Structure: Every node contains a Data payload and a Pointer to its successor.
  • Use Cases: Best for applications with frequent additions/removals where random-access speed is not critical.

Chapter 03

Queues — The Amusement Park Line

Enqueue O(1) Dequeue O(1)

The Queue is best understood as the line for an amusement park ride. The rule is sacred, unbreakable, and absolute: First-In, First-Out (FIFO). The first person to join is the first person to experience the ride.

The Queue maintains two critical positions: the Front (Head), where the next passenger is ready to exit, and the Rear (Tail), where all new patrons must join. Joining at the back (Enqueue) and leaving from the front (Dequeue) are both instant O(1) operations — no matter how long the line stretches.

This simplicity is the Queue's strength. It perfectly models tasks requiring strict chronological order and fairness: task scheduling, print jobs, network buffering, and Breadth-First Search algorithms.

💡 Analogy: A queue is like a printer spooling tasks — the first job sent is the first job printed. No cutting in line.

In-Depth Learnings

  • Core Principle: FIFO (First-In, First-Out) — strict chronological fairness of access.
  • Enqueue: Insert at the Rear/Tail — O(1).
  • Dequeue: Remove from the Front/Head — O(1).
  • Structure: Requires tracking two pointers: Front and Rear.
  • Use Cases: Task scheduling, printer queues, BFS graph traversal, data buffering.
  • Gotcha: A naive array implementation causes O(n) dequeue due to element shifting — use a circular array or linked list instead.

Chapter 04

Searching — The Library Hunt

Linear O(n) Binary O(log n)

Searching is the fundamental quest to locate a single desired item within a vast collection. Our library illustrates the difference between efficiency and brute force.

Linear Search — the disorganized librarian. Books are scattered randomly, utterly unsorted. To find the scroll, you check every title, shelf by shelf. In the worst case it's the very last item: O(n). Simple to implement, but terrible for large libraries.

Binary Search — the master strategist. One prerequisite: all books must be sorted. Walk to the middle, ask "is the target before or after?" Instantly discard half the library. Repeat. A million-book library can be searched in just 20 steps — O(log n).

💡 Linear Search is like scrolling an unordered contact list (O(n)). Binary Search is like using a phonebook — open the middle, go left or right (O(log n)).

In-Depth Learnings

  • Linear Search: O(n) — checks up to N elements in the worst case. Works on unsorted data.
  • Binary Search Pre-requisite: Data must be sorted before searching begins.
  • Binary Search: O(log n) — halves the search space at every step. Extremely fast on large datasets.
  • Trade-off: Binary search is fast, but the initial sort costs up to O(n log n).
  • Use Cases: Binary search for large, static, sorted databases. Linear search for small or unsorted lists.

Chapter 05

Sorting — Arranging the Blueprints

Bubble Sort O(n²) Merge Sort O(n log n)

Sorting is the art of imposing order on chaos. The true complexity lies not in the final ordered stack, but in the journey taken to achieve it.

The architect using Bubble Sort compares two adjacent blueprints and swaps them if they're in the wrong order, then moves to the next pair — iterating again and again until the largest bubble to the top. Multiple full passes across nearly all pairs gives a massive O(n²) complexity. Fine for a cottage; an eon for a skyscraper.

Then there's Merge Sort. The architect divides the stack into smaller and smaller sub-stacks until each contains just one item (trivially sorted), then merges them back in perfect order. This "Divide and Conquer" strategy scales beautifully to O(n log n) — the gold standard for massive datasets.

💡 O(n²) is like fixing every misplaced card one swap at a time. O(n log n) is like splitting the deck in half, sorting each half, then merging back together.

In-Depth Learnings

  • Quadratic Sorts: Bubble, Insertion, and Selection Sort — O(n²) average. Best only for very small arrays.
  • Log-Linear Sorts: Merge Sort and Quick Sort — O(n log n) average. Workhorses of industry.
  • Space Complexity: Merge Sort requires O(n) auxiliary space; Quick Sort (in-place) uses only O(log n) stack space.
  • Stability: A sort is stable if it preserves the relative order of equal elements — critical for multi-key database sorting.
  • Adaptive Sorting: Insertion Sort shines on nearly-sorted input, approaching O(n).

Maximum Subarray Sum (Kadane's)

Find the contiguous subarray (containing at least one number) which has the largest sum.

Use Kadane's Algorithm — a single-pass greedy/DP approach. Track currentMax (best sum ending here) and globalMax (best seen so far). At each element: currentMax = max(num, currentMax + num). Time Complexity: O(n). Space: O(1).
Meta Easy

Reverse a Linked List

Given the head of a singly linked list, reverse the list and return the new head.

Iterative approach using three pointers: prev = null, curr = head, next = null. Loop: store next = curr.next, reverse the pointer curr.next = prev, advance both. Return prev when done. Time: O(n). Space: O(1).
Amazon Easy

Two Sum

Return the indices of the two numbers in an array that add up to a specific target.

Use a Hash Map. For each element num at index i, look up target - num in the map. If found, return both indices. Otherwise, store map[num] = i. Single pass lookup is O(1) per step. Time: O(n). Space: O(n).
